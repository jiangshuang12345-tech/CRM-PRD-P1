# CRM 运营平台 · 一期「基础查询」PRD

> **本版本重大变更（重新分期）**
> 1. **渠道体系整体后移至二期**：渠道管理及「业务线 · 渠道 · 渠道 code」精准归因不在一期范围。一期用户归因降级为 **应用商店来源（App Store / Google Play）+ 国家（注册 IP）** 粗归因。
> 2. **用户状态细化为 5 态**：注册 / 体验中 / 体验逾期 / 付费 / 付费逾期（原「流失」更名为「付费逾期」，新增「体验逾期」）。
> 3. **多语言默认英文**：开放 英 / 韩 / 阿 / 中 四种，默认 English。
> 4. **注册时间口径为 UTC**。
>
> 说明：渠道管理、用户中心·二期、商品包、优惠券、落地页、系统配置/角色权限均属**二期**，不纳入本文档（仅在「与二期衔接」处简述）。

# 背景

- Dino English 面向海外多业务线（韩国 / 沙特 / 马来 / 越南 等）开展投放与运营。当前业务依赖人工散表查询用户与订单、归因口径不统一、促销配置依赖研发。
- 一期先解决「用户、订单查得到、看得清」，让客服 / 运营 / 市场自助完成日常查询；此时**尚无渠道分级体系**，用户归因仅做粗粒度（应用商店来源 + 国家）：

| 业务场景 | 现在怎么做 | 当前问题 | 一期是否解决 |
| --- | --- | --- | --- |
| 查用户 | 找研发拉数 / 翻多张散表 | 非手机号注册（谷歌/FB/AppID）的用户难以查到；查询慢、口径乱 | ✅ 解决（自助查询）|
| 查订单 | 找研发或财务导出对账 | 客服无法自助核对实付金额、支付方式、支付时间、有效期 | ✅ 解决（自助查询）|
| 渠道归因 | 渠道用线下表格、code 人工拼 | 归因口径不统一、无唯一 code | ⏳ 二期（一期先按应用商店 + 国家粗归因）|

# 目标

- 一期定位：**先把「用户、订单查得到、看得清」做扎实**，让市场与客服同学自助完成日常查询，替代线下散表。
- 一期范围（两个查询模块 + 一项平台能力）：
  1. **用户中心（一期）**：多条件检索用户，支持 5 种注册方式与登录账号展示；归因仅按 **应用商店来源（App Store / Google Play）+ 国家**；可维护基础资料。
  2. **订单信息查询（订单中心）**：多条件检索订单，查看商品、归属用户、金额、支付信息与有效期（一期只查不改）。
  3. **平台能力**：多语言（英 / 韩 / 阿 / 中，默认英文）切换；登录限 `@dinoai.ai` 邮箱。
- 衡量指标：
  - 用户 / 订单查询自助率 ≥ 90%
  - （渠道归因覆盖率 ≥ 95% 为**二期**指标，一期不承诺）

# 名词约定 / 颜色图例

- **注册渠道（一期口径）**：一期仅区分应用商店来源 **App Store / Google Play**（无渠道树、无业务线、无渠道 code）。
- **国家**：用户注册时 IP 对应的国家，一期用于替代二期的「业务线」展示与筛选。
- **注册方式**：用户注册时使用的账号类型，共 5 种（谷歌邮箱 / Facebook / kakao / 手机号 / AppID）。
- **登录账号**：用户登录使用的账号值（邮箱 / Facebook / kakao ID / AppID / 手机号；手机号场景已并入登录账号）。
- **用户状态**：系统按商品包有效期与续订情况自动流转的状态（注册 / 体验中 / 体验逾期 / 付费 / 付费逾期），不可手工编辑。
- **业务线**：按国家 / 地区划分的运营单元；一期仅**订单中心**使用，用户中心一期改用「国家」（业务线 ↔ 渠道精准归因为二期能力）。
- **【待确认】**：需与研发 / 投放侧拉齐口径的内容，见各模块标注与附录清单。

## 一、用户中心（一期）（P0）

客服 / 运营按多条件检索用户，查看基本信息、注册方式与登录账号、状态与有效期，并可维护基础资料。**一期暂无渠道体系**：注册来源仅区分应用商店（App Store / Google Play），归属国家取注册时 IP 对应国家。

**用户故事**：作为客服同学，我希望输入用户ID / 姓名 / 登录账号即可查到用户，并看到其注册方式、登录账号、注册渠道（App Store / Google Play）、国家、注册时间、到期时间与当前状态——即使用户用谷歌邮箱 / Facebook / AppID 等**非手机号方式**注册也能查到。

> **一期 vs 二期（重要）**
> 一期用户中心**不展示**「业务线 / 渠道 / 渠道 code」，仅用 **注册渠道（App Store / Google Play）+ 国家** 做粗粒度归因；待二期渠道管理上线后，升级为「用户中心·二期」，按「业务线 · 渠道 · 渠道 code」精准归因。

| 一级模块 | 二级模块 | 描述（产品逻辑） | 优先级 |
| --- | --- | --- | --- |
| 查询 | 关键字搜索 | 支持 用户ID / 姓名 / 登录账号 **模糊搜索**（登录账号即覆盖手机号场景） | P0 |
| 查询 | 条件筛选 | 可按 **注册渠道（App Store / Google Play）**、**国家**、用户状态（注册/体验中/体验逾期/付费/付费逾期）筛选 | P0 |
| 列表 | 列表 + 分页 | 1. 展示基本信息字段（见数据字典）<br>2. 支持分页与每页条数<br>3. 列表实时按条件过滤 | P0 |
| 列表 | 注册方式 / 登录账号展示 | 1. 展示「注册方式」「登录账号」列；支持 5 种注册方式<br>2. 姓名列展示 App 内姓名（`Student_Name`）<br>3. 注册渠道列仅展示 App Store / Google Play；另有「国家」列 | P0 |
| 列表 | 到期时间 / 最近修改人 | 1. 展示会员 / 商品包到期时间，无则显示「—」<br>2. 展示最近一次资料修改的操作人，无则显示「—」 | P0 |
| 编辑 | 修改基本信息 | 1. 可编辑：姓名（`Student_Name`）/ 性别 / 出生日期<br>2. **国家只读、不可修改**<br>3. 点「修改信息」→ 弹窗编辑 → 保存，系统记录「最近修改人」<br>4. 用户状态为系统流转，不可编辑 | P1 |

**数据字典（列表字段）**

| 字段 | 说明 | 可编辑 |
| --- | --- | --- |
| 用户ID | 用户唯一标识（长数字ID） | 否 |
| 学生姓名 | 单一姓名字段，取 App 内 `Student_Name` | 可编辑 |
| 注册方式 | 谷歌邮箱 / Facebook / kakao / 手机号 / AppID | 否 |
| 登录账号 | 邮箱 / Facebook / kakao ID / AppID / 手机号 | 否 |
| 注册渠道（一期口径） | 仅区分应用商店来源：App Store / Google Play（一期无渠道树） | 否 |
| 国家（一期口径） | 注册时 IP 对应的国家（替代二期的「业务线」展示） | 否 |
| 注册时间 | UTC 时间 | 否 |
| 到期时间 | 会员 / 商品包到期时间，无则显示「—」 | 否 |
| 用户状态 | 注册 / 体验中 / 体验逾期 / 付费 / 付费逾期 | 否（系统流转） |
| 最近修改人 | 最近一次资料修改的操作人，无则显示「—」 | 否（系统记录） |

**注册方式与登录账号映射**

| 注册方式 | 登录账号示例 | 是否带手机号 |
| --- | --- | --- |
| 谷歌邮箱 | 邮箱地址 | 否 |
| Facebook | 邮箱（可拿到时） | 否 |
| AppID | 邮箱（可拿到时） | 否 |
| kakao | 手机号 | 是 |
| 手机号 | 手机号 | 是 |

**用户状态定义（系统自动流转，不可手工编辑）**

| 用户状态 | 定义 | 判定口径 |
| --- | --- | --- |
| 注册 | 已注册未付费 | 完成注册，从未付费 |
| 体验中 | 已付费 7 天体验商品包，且在有效期内 | 持有体验包 且 当前时间 ≤ 体验有效期 |
| 体验逾期 | 体验包已到期，且未转为正式付费 | 曾持有体验包 且 当前时间 > 体验有效期 且 未付费正式包 |
| 付费 | 付费正式商品包，且在有效期内（含到期自动续订） | 持有正式包 且 当前时间 ≤ 有效期 |
| 付费逾期 | 付费正式商品包，已过有效期且未续订（原「流失」） | 曾持有正式包 且 当前时间 > 有效期 且 未续订 |

> 状态随有效期推移与续订结果变化：体验到期未转正式即为「体验逾期」；正式包到期自动续订成功仍为「付费」，到期未续订转为「付费逾期」。**【待确认】** 状态计算所需的「有效期 / 续订状态」字段由订单 / 履约侧提供。

**关键交互流程**

1. 输入关键字（用户ID/姓名/登录账号）或选择 注册渠道（App Store/Google Play）/ 国家 / 用户状态 → 列表实时过滤。
2. 查看「注册方式」「登录账号」列定位用户来源；非手机号方式登录账号为邮箱 / 第三方账号。
3. 点「修改信息」→ 弹窗编辑 姓名（`Student_Name`）/ 性别 / 出生日期（国家只读）→ 保存，系统记录「最近修改人」。

## 二、订单信息查询（订单中心）（P0）

按多条件检索订单，查看商品、归属用户、金额与支付信息，支撑客服对账与退款核查。**一期仅做查询，不含创建 / 退款等写操作。**

**用户故事**：作为客服 / 财务同学，我希望按订单ID / 用户ID / 商品检索订单，并能按业务线、订单状态、支付方式筛选，快速核对实付金额、支付时间与有效期。

| 一级模块 | 二级模块 | 描述（产品逻辑） | 优先级 |
| --- | --- | --- | --- |
| 查询 | 关键字搜索 | 支持 订单ID / 用户ID / 商品名称 **模糊搜索** | P0 |
| 查询 | 条件筛选 | 可按 **业务线**、订单状态（待支付/已支付/已退款/已取消）、支付方式 筛选 | P0 |
| 列表 | 列表 + 分页 | 1. 展示 原价 / 实付 / 币种 / 支付时间 / **有效期到期时间** 等字段<br>2. 支持分页 | P0 |
| 范围约束 | 只读 | 一期不含创建 / 退款 / 取消等写操作；退款等待权限体系（二期）就绪后再开放 | P0 |

> 业务线「全局共享筛选」在二期与用户中心·二期联动；一期订单中心可独立按业务线筛选。

**数据字典（列表字段）**

| 字段 | 说明 |
| --- | --- |
| 订单ID | 订单唯一标识，如 `DN2026061800001` |
| 商品名称 / 用户ID | 所购商品 + 归属用户 |
| 所属业务线 | 按归属用户的业务线展示 |
| 用户状态 / 订单状态 | 用户状态 + 订单状态双标签 |
| 原价 / 实际付款金额 | 含币种，按业务线本地币种或 USD 展示 |
| 支付方式 | App Store / Google Play / Stripe / PayPal |
| 成功支付时间 | 未支付显示 `—` |
| 有效期到期时间 | 订单对应权益的到期时间，无则显示「—」 |

# 与二期的衔接（仅说明，不在一期范围）

为便于研发预留扩展，列出一期数据将如何被二期复用（**以下均为二期实现**）：

| --- | --- |
