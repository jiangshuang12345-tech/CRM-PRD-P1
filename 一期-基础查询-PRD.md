# CRM 运营平台 · 一期「基础查询」PRD

> 本版本同步整体规划 PRD 更新：① 渠道树新增最顶层「业务线」层级、code 改为 7 位随机码；② 用户中心移除手机号/国家码列、新增「到期时间」「最近修改人」列、搜索改为 用户ID/姓名/登录账号；③ 订单中心新增「所属业务线」「有效期到期时间」列及业务线筛选；④ 平台新增多语言（中/英/韩/阿）、业务线全局共享筛选。

# 背景

- Dino English 面向海外多业务线（韩国 / 沙特 / 马来 / 越南 等）开展投放与运营。当前业务依赖人工散表查询用户与订单、渠道归因口径不统一、促销配置依赖研发。
- 运营 / 客服 / 市场同学每天处理的核心动作是「查渠道、查用户、查订单」，但当前链路存在如下问题：

| 业务场景 | 现在怎么做 | 当前问题 |
| --- | --- | --- |
| 市场渠道管理 | 各投放渠道用线下表格记录，code 人工拼 | 渠道分级口径不统一、code 无唯一标准，用户注册无法准确归因到业务线与来源 |
| 查用户 | 找研发拉数 / 翻多张散表 | 非手机号注册（谷歌/FB/AppID）的用户难以查到；查询慢、口径乱 |
| 查订单 | 找研发或财务导出对账 | 客服无法自助核对实付金额、支付方式、支付时间、有效期 |

- 整体规划 PRD（含交互原型）地址：<https://jiangshuang12345-tech.github.io/CRM-PRD/#mapping>
- 在线原型：<https://jiangshuang12345-tech.github.io/CRM/#/channels>（纯前端 Mock，登录仅支持 `@dinoai.ai` 邮箱，验证码直接显示在页面提示中）

# 目标

- 一期定位：**先把「查得到、看得清、能归因」做扎实**，让市场与客服同学自助完成日常查询，替代线下散表。
- 三个核心能力 + 一项平台能力：
  1. **渠道管理**：渠道结构标准化（**业务线 → 渠道类型 → 一/二/三级**），任意层级生成唯一渠道 code，作为用户注册归因的唯一口径。
  2. **用户基本信息查询**：多条件检索用户，支持 5 种注册方式与登录账号展示，可维护基础资料。
  3. **订单信息查询**：多条件检索订单，查看商品、归属用户、金额、支付信息与有效期（一期只查不改）。
  4. **平台能力**：多语言（中 / 英 / 韩 / 阿）切换；业务线筛选全局共享联动；登录限 `@dinoai.ai` 邮箱。
- 衡量指标：
  - 用户 / 订单查询自助率 ≥ 90%
  - 渠道归因覆盖率 ≥ 95%

# 名词约定 / 颜色图例

- **业务线**：按国家 / 地区划分的运营单元（韩国 / 沙特 / 马来 / 越南 / 其他 …）。本期为**渠道树最顶层节点**，且支持自定义新增。
- **渠道 code（channel_code）**：投放与归因使用的唯一标识，由「渠道管理」生成（7 位随机字母数字），是渠道 ↔ 用户 ↔ 订单的连接键。
- **注册方式**：用户注册时使用的账号类型，共 5 种（谷歌邮箱 / Facebook / kakao / 手机号 / AppID）。
- **登录账号**：用户登录使用的账号值（邮箱 / Facebook / kakao ID / AppID / 手机号；手机号场景已并入登录账号）。
- **用户状态**：系统按商品包有效期与续订情况自动流转的状态（注册 / 体验 / 付费 / 流失），不可手工编辑。
- **全局业务线筛选**：用户中心、订单中心共享同一业务线筛选，一处切换全局联动。
- **【待确认】**：需与投放侧 / 研发拉齐口径的内容，见各模块标注与附录清单。

# 需求描述

一期围绕「查询」一条主线，三个模块通过 `渠道 code` 串联归因关系，并在平台层提供多语言与全局业务线筛选：

## 一、渠道管理（P0）

维护标准化的渠道分级结构，并在任意层级生成实际投放使用的渠道 code，作为用户注册归因的唯一口径。

**用户故事**：作为市场投放同学，我希望按「业务线 → 渠道类型 → 一级 → 二级 → 三级」维护渠道，并为具体投放位生成唯一 code，以便用户注册时能准确归因到业务线与来源。

> **本期更新点（重点实现）**
> 1. 渠道树新增**最顶层「业务线」层级**，结构调整为 **业务线 → 渠道类型 → 一级 → 二级 → 三级**。
> 2. 入口按钮由「新增渠道类型」改为「**新增业务线**」（在业务线下再新增渠道类型）。
> 3. 渠道 code 生成规则改为 **7 位随机字母数字**（如 `K2000Gh`），**不再带类型前缀**。

## 二、用户基本信息查询（P0）

客服 / 运营按多条件检索用户，查看基本信息、注册方式与登录账号、注册归因与状态，并可维护基础资料。

**用户故事**：作为客服同学，我希望输入用户ID / 姓名 / 登录账号即可查到用户，并看到其注册方式、登录账号、业务线、注册渠道、注册时间、到期时间与当前状态——即使用户用谷歌邮箱 / Facebook / AppID 等**非手机号方式**注册也能查到。

> **本期更新点（重点实现）**
> 1. 列表**移除「手机号」「注册地国家码」两列**（手机号已并入登录账号：kakao / 手机号 方式的登录账号即手机号）。
> 2. 「注册渠道」列改为展示 **业务线 · 渠道**。
> 3. **新增「到期时间」「最近修改人」两列**。
> 4. 关键字搜索维度调整为 **用户ID / 姓名 / 登录账号**（移除手机号独立维度）。
> 5. 业务线筛选改为**全局共享**（与订单中心联动）。

**用户状态定义（系统自动流转，不可手工编辑）**

| 用户状态 | 定义 | 判定口径 |
| --- | --- | --- |
| 注册 | 已注册未付费 | 完成注册，从未付费 |
| 体验 | 已付费 7 天体验商品包，且在有效期内 | 持有体验包 且 当前时间 ≤ 体验有效期 |
| 付费 | 付费正式商品包，且在有效期内（含到期自动续订） | 持有正式包 且 当前时间 ≤ 有效期 |
| 流失 | 付费正式商品包，已过有效期且未续订 | 曾持有正式包 且 当前时间 > 有效期 且 未续订 |

> 状态随有效期推移与续订结果变化：正式包到期自动续订成功仍为「付费」，到期未续订转为「流失」。**【待确认】** 状态计算所需的「有效期 / 续订状态」字段由订单 / 履约侧提供。

**关键交互流程**

1. 输入关键字（用户ID/姓名/登录账号）或选择 业务线（全局）/ 用户状态 → 列表实时过滤。
2. 查看「注册方式」「登录账号」列定位用户来源；非手机号方式登录账号为邮箱 / 第三方账号。
3. 点「修改信息」→ 弹窗编辑 本地名 / 性别 / 出生日期 / 业务线 → 保存，系统记录「最近修改人」。

**用户故事**：作为客服 / 财务同学，我希望按订单ID / 用户ID / 商品检索订单，并能按业务线、订单状态、支付方式筛选，快速核对实付金额、支付时间与有效期。

> **本期更新点（重点实现）**
> 1. 订单列表**新增「所属业务线」「有效期到期时间」两列**。
> 2. 新增**业务线筛选**（与用户中心全局共享）。
> 3. 关键字搜索维度由 学生ID 调整为 **用户ID**。

| 一级模块 | 二级模块 | 描述（产品逻辑） | 优先级 |

| 能力 | 说明 |

**状态枚举**

- 订单状态：待支付 / 已支付 / 已退款 / 已取消
- 用户状态：注册 / 体验 / 付费 / 流失

**待确认事项（开发前拉齐）**

1. 渠道 code 生码规则（大小写、是否可重新生码、唯一性校验策略）与投放侧最终口径。
2. 用户 / 订单数据来源：新建库 vs 对接现有业务库。
3. 用户状态、用户「到期时间」、订单「有效期到期时间」所需的有效期 / 续订状态字段由谁提供、实时计算还是定时刷新。
4. 「最近修改人」记录范围（仅资料修改，还是含其他操作）与是否需要完整操作日志 / 审计（为后续权限体系预留）。
5. 新增业务线的币种 / 国家码映射补充约定。
